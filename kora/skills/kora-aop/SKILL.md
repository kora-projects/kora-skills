# kora-aop — compile-time aspect-oriented programming

Read this first when:
- adding method validation with `@Valid`/`@Validate` and constraint annotations (`@NotEmpty`, `@Pattern`, `@Range`),
- wiring method logging with `@Log` for arguments/results or `@Mdc` for structured logging context,
- implementing resilience patterns with `@CircuitBreaker`, `@Retry`, `@Timeout`, `@Fallback`,
- scheduling background tasks with `@ScheduleAtFixedRate`, `@ScheduleWithFixedDelay`, `@ScheduleWithCron`,
- caching method results with `@Cacheable`, `@CachePut`, `@CacheInvalidate` (Caffeine or Redis),
- debugging aspects not applying (check: class is non-final, method is non-private, module is plugged in).

## Quick Start

### 1. Add the module

```groovy
// Validation
implementation "ru.tinkoff.kora:validation-module"

// Logging (usually already on classpath)
implementation "ru.tinkoff.kora:logging-common"

// Resilience
implementation "ru.tinkoff.kora:resilient-kora"

// Scheduling (JDK or Quartz)
implementation "ru.tinkoff.kora:scheduling-jdk"        // fixed-rate, fixed-delay, once
implementation "ru.tinkoff.kora:scheduling-quartz"     // cron expressions

// Caching (Caffeine or Redis)
implementation "ru.tinkoff.kora:cache-caffeine"
implementation "ru.tinkoff.kora:cache-redis"
```

### 2. Plug into Application

```java
@KoraApp
public interface Application extends 
    ValidationModule,
    LoggingModule,
    ResilientModule,
    SchedulingJdkModule,
    CaffeineCacheModule 
{ }
```

### 3. Annotate your class (NOT final!)

```java
@Component
public class OrdersService {        // ← must NOT be final (Java) / must be open (Kotlin)
    
    @Validate
    public Order create(@Valid CreateOrder body) {
        return new Order(/* ... */);
    }
    
    @Log
    public Order get(Long id) {
        return repo.find(id);
    }
}
```

### 4. Use assets templates

See `assets/` for ready-to-use templates (Java + Kotlin):
- `ValidatedDto.java.template` / `.kt.template` — DTO with validators
- `ValidatedService.java.template` / `.kt.template` — service with `@Validate`
- `ResilientService.java.template` / `.kt.template` — `@Timeout` + `@Retry` + `@CircuitBreaker` + `@Fallback`
- `ScheduledJobs.java.template` / `.kt.template` — JDK + Quartz schedulers
- `OrderCache.java.template` / `.kt.template` — `@Cacheable` + `@CachePut` + `@CacheInvalidate`

## The unifying mechanism

Every cross-cutting feature in this skill is implemented the same way: a **compile-time aspect** generated by Kora's annotation processor wraps the target method. There is no runtime proxy, no `cglib`, no reflection-based interception. The processor emits a generated subclass that overrides each annotated method with the additional logic (validation check, log statement, circuit-breaker call, cache lookup, scheduler hook).

This has three practical consequences you must internalize:

1. **The class must be subclassable.**
   - Java: the class containing the annotated method must **not** be `final` (and the method must **not** be `private`).
   - Kotlin: the class must be `open` (or `abstract`).
   Forget this and the compiler errors with "aspect class must be non-final / open".

2. **Annotations stack and compose.** Multiple aspects on the same method are applied in declaration order, top to bottom. The order matters: `@Timeout` outside `@Retry` means "the whole retry budget has this timeout"; `@Retry` outside `@Timeout` means "each attempt has this timeout".

3. **No runtime registration.** If a third-party annotation is referenced but the supporting module isn't on the classpath, the annotation is silently ignored at compile time — you get a method without the aspect and no warning. Always plug the matching `*Module` into `@KoraApp`.

## Validation — quick patterns

```java
@Valid                                    // generates Validator<CreateOrder>
public record CreateOrder(
    @NotBlank @Size(max = 64) String sku,
    @Range(from = 1, to = 1000) int quantity,
    @Nullable String note
) {}

@Component
public class OrdersService {                            // not final → aspect can subclass
    @Validate                                           // validates args + return
    public Order create(@Valid CreateOrder body) {
        return new Order(/* ... */);
    }
}
```

- `@Valid` on a record → `Validator<T>` is registered as a Kora component, injectable anywhere.
- `@Validate` on a method enables runtime validation of args / return.
- Annotations on parameters: `@NotEmpty`, `@NotBlank`, `@Pattern(regex=...)`, `@Range(from=, to=)`, `@Size(min=, max=)`.
- `@Nullable` parameter is exempt from the implicit `@NotNull`.
- Validation failure throws `ViolationException`. Catch in your global `ErrorInterceptor` (see `kora-server`).

Full detail: `references/validation.md`.

## Method logging — quick patterns

```java
@Component
public class OrdersService {
    @Log                                                // log args on enter, result on exit
    @Mdc(key = "tenant", value = "${request.tenantId}") // attach to MDC for the duration
    public Order create(@Mdc CreateOrder body) {        // body field/value lands in MDC too
        ...
    }

    @Log.out                                            // log only the result
    public Order get(@Log.off UUID id) { ... }          // don't include `id` in logs
}
```

- `@Log` = `@Log.in` + `@Log.out` (log args + return).
- Verbosity by log level: at TRACE/DEBUG, includes data; at INFO, only the boundary marker.
- `@Mdc` puts entries into MDC for the duration of the call; `global = true` keeps them on the thread.
- `@Log.off` on a parameter suppresses logging just that value (PII, secrets).

> **Use Kora's MDC, never SLF4J's.** The annotation `@Mdc` is `ru.tinkoff.kora.logging.common.annotation.Mdc`. Imperative MDC is `ru.tinkoff.kora.logging.common.MDC.put(...)` — **never** `org.slf4j.MDC.put(...)`. SLF4J's `MDC` doesn't flow through `KoraAsyncAppender` and isn't rendered by `ConsoleTextRecordEncoder` — values silently vanish. IDE auto-import gravitates to SLF4J; verify the import every time. See `references/logging-aspect.md` for the full callout.

Full detail: `references/logging-aspect.md`.

## Resilience — quick patterns

```java
@Component
public class InventoryService {

    @Timeout("inventory.check")                      // outer aspect
    @Retry("inventory.check")                        // inner aspect
    @Fallback(value = "inventory.check", method = "checkFallback()")
    public Stock check(String sku) {
        return inventoryClient.lookup(sku);
    }

    protected Stock checkFallback() {
        return Stock.UNKNOWN;
    }
}
```

```hocon
resilient {
  timeout.inventory.check        { duration = "1s" }
  retry.inventory.check          { delay = "100ms", attempts = 3, delayStep = "100ms" }
  circuitbreaker.inventory.check { slidingWindowSize = 100, minimumRequiredCalls = 10,
                                   failureRateThreshold = 50, waitDurationInOpenState = "25s",
                                   permittedCallsInHalfOpenState = 15 }
}
```

- All four annotations take a **configuration name**. Settings live under `resilient.{circuitbreaker|retry|timeout|fallback}.<name>`.
- Each has a `default` block whose values are inherited by every named instance.
- `@Fallback(method = "fallbackMethod()")` — the method must be in the same class, signature compatible.
- Imperative APIs: `CircuitBreakerManager`, `RetryManager`, `TimeoutManager`, `FallbackManager` — inject and call by name.
- Predicates (`CircuitBreakerPredicate`, `RetryPredicate`, `FallbackPredicate`) decide which exceptions count; register a `@Component` returning a name, refer to it via `failurePredicateName = "MyPredicate"` in config.

**Aspect order matters.** Outer annotation runs first. With `@Timeout` on top and `@Retry` inside: the timeout caps the **entire** retry budget. Flip them: each retry attempt gets its own timeout.

Full detail: `references/resilient.md`.

## Scheduling — quick patterns

```java
@Component
public class OrdersCleanupJob {

    @ScheduleAtFixedRate(initialDelay = 30, period = 60, unit = ChronoUnit.SECONDS)
    public void everyMinute() { ... }

    @ScheduleWithFixedDelay(config = "jobs.cleanup")        // values from config
    public void afterPrevious() { ... }

    @ScheduleOnce(delay = 5, unit = ChronoUnit.MINUTES)     // single run after startup
    public void warmup() { ... }

    @ScheduleWithCron("0 0 3 * * ?")                        // Quartz only — daily at 03:00
    public void nightlyReport() { ... }
}
```

```hocon
jobs.cleanup { initialDelay = "30s", delay = "1m" }
scheduling.threads = 4                # shared ScheduledExecutorService for JDK scheduling
```

- **JDK scheduling** (`scheduling-jdk` + `SchedulingJdkModule`) covers `@ScheduleAtFixedRate`, `@ScheduleWithFixedDelay`, `@ScheduleOnce`. Backed by `ScheduledExecutorService`.
- **Quartz** (`scheduling-quartz` + `QuartzModule`) adds `@ScheduleWithCron` and `@ScheduleWithTrigger` (custom `Trigger` injection by tag). Configure via standard Quartz `org.quartz.*` properties under `quartz.{...}`.
- Annotation parameters or `config = "path"`: when `config` is set, values come from the config file (overrides annotation). HOCON durations (`"50ms"`, `"5s"`, `"1h"`).
- For Quartz only: `@DisallowConcurrentExecution`, `@PersistJobDataAfterExecution`.
- Graceful shutdown: scheduled methods should check `Thread.currentThread().isInterrupted()` to exit early.

Full detail: `references/scheduling.md`.

## Caching — quick patterns

```java
// 1. Declare a typed cache interface
@Cache("orders.cache")
public interface OrderCache extends CaffeineCache<UUID, OrderDto> {}

// 2. Use it via annotations
@Component
public class OrdersService {
    @Cacheable(OrderCache.class)
    public OrderDto get(UUID id) { ... }                    // read-through

    @CachePut(OrderCache.class)
    public OrderDto save(UUID id, OrderDto dto) { ... }     // write-through

    @CacheInvalidate(OrderCache.class)
    public void delete(UUID id) { ... }                     // evict by key

    @CacheInvalidate(value = OrderCache.class, invalidateAll = true)
    public void clearAll() { ... }
}
```

```hocon
orders.cache {                                              # matches @Cache("orders.cache")
  expireAfterWrite  = "10m"
  expireAfterAccess = "5m"
  maximumSize       = 10000
}
```

- `@Cache("config.path")` declares the cache interface. It must extend `CaffeineCache<K,V>` or `RedisCache<K,V>`.
- Cache key is built from method arguments in order. `parameters = {"arg1", "arg2"}` selects subset / reordering.
- Composite key: declare a `record Key(...)` inside the cache interface and extend `CaffeineCache<MyCache.Key, V>`.
- Complex argument → simple key: implement `CacheKeyMapper<K, ArgType>` and apply `@Mapping(MyMapper.class)` next to `@Cacheable`.
- `LoadableCache<K,V>` combines get + put without aspects — call `cache.asLoadable(loader)` in a `@KoraApp` factory.
- For Redis caching, set `lettuce.uri = ${REDIS_URL}` and use `RedisCache<K,V>` in the cache interface.

Full detail: `references/cache.md`.

## Common pitfalls

- **Forgetting `open`/non-final.** Most common error. Kotlin: mark the class `open`. Java: drop `final`. The compile error is explicit, but easy to miss in IDE quick-fix suggestions.
- **Annotating a static method or private method.** Aspects can't subclass these. Make instance + at least package-private.
- **Stacking aspects in the wrong order.** `@Timeout` outside `@Retry` is "whole retry budget timeout"; flip for "per-attempt timeout". `@Cacheable` outside `@CircuitBreaker` skips the breaker on cache hits — usually what you want. Test the ordering with intentional failure.
- **Validation throwing during HTTP request → returning 500.** Without a global `ErrorInterceptor` that catches `ViolationException`, validation failures become 500s. Map them to 400 in your `ErrorInterceptor` (standard pattern: `@Component class ErrorInterceptor implements ru.tinkoff.kora.http.server.common.HttpServerInterceptor`).
- **Resilient annotation referencing a non-existent config name.** Compile succeeds (the name is just a string), but `resilient.<kind>.<name>` is required at startup — graph fails with "value not found". Always create the matching block (or rely on `default` and override only what differs).
- **`@Log` on a method returning `Mono`/`Flux`/`CompletionStage` without the relevant dependencies.** `logging-aspect` supports those signatures, but you must include `reactor-core` or `kotlinx-coroutines-core` in implementation deps.
- **Importing `org.slf4j.MDC` instead of `ru.tinkoff.kora.logging.common.MDC`.** SLF4J's stock `MDC` doesn't propagate through `KoraAsyncAppender` and isn't rendered by `ConsoleTextRecordEncoder` — values silently vanish from log output. IDE auto-import picks SLF4J by default; verify the import explicitly every time you write `MDC.put(...)`. The same applies to the annotation: `@Mdc` must be `ru.tinkoff.kora.logging.common.annotation.Mdc` (there is no SLF4J equivalent).
- **Cache key ambiguity.** When multiple methods on the same cache use different signatures, ensure their argument projections collide on neither. `parameters = {...}` is the explicit fix.
- **Redis cache without `keyPrefix`.** Each `RedisCache` config requires `keyPrefix` to avoid collisions across cache instances on the same Redis DB. Default behavior is to fail at startup if missing.
- **Quartz `@ScheduleWithCron` without `QuartzModule`.** `scheduling-jdk` doesn't support cron. If you need cron, switch to or add `scheduling-quartz`.

## Don't do

- **Don't annotate `final` classes or `private` methods.** Compile-time AOP requires subclassing. Use `open` (Kotlin) or non-final (Java).
- **Don't stack `@Log` + `@Validate` on hot-path methods without measuring overhead.** Each aspect adds bytecode; profile before applying both on high-QPS methods.
- **Don't use `@Cacheable` without `@CacheInvalidate` strategy.** Stale cache is worse than no cache — always define eviction (TTL, `@CacheInvalidate`, or `maximumSize`).
- **Don't catch `ViolationException` in business logic.** Let it propagate to `ErrorInterceptor` for consistent 400 mapping.
- **Don't apply `@Retry` to idempotency-sensitive operations without `@Idempotent` guard.** Retry + non-idempotent = duplicate charges/orders.
- **Don't schedule `@ScheduleAtFixedRate` with period < execution time.** You'll get concurrent runs — use `@ScheduleWithFixedDelay` or `@DisallowConcurrentExecution`.
- **Don't forget `@KoraApp` module registration.** Annotation without module = silently ignored at compile time.
- **Don't use SLF4J `MDC` directly.** Use `ru.tinkoff.kora.logging.common.MDC` for proper async context propagation.

## Best Practices

1. **Start with `@Validate` on DTOs, not services.** Validate at the boundary (HTTP/gRPC payload) before it enters your service layer. Use `@Validate` on service methods only for cross-method invariants.

2. **Use `@Log.in` for fire-and-forget, `@Log` for request/response.** Full `@Log` on high-QPS methods can flood logs — prefer `@Log.in` for side effects and `@Log.result` when only the return value matters.

3. **Resilience: prefer `default` config, override per-method.** Define `resilient.retry.default`, `resilient.circuit-breaker.default` once, then reference by name. Only override specific keys (e.g., `attempts = 5` for critical calls).

4. **Circuit breaker: set `requestVolumeThreshold`.** Without it, CB opens on a few failures during low traffic — set `requestVolumeThreshold = 10` to require minimum call volume before tripping.

5. **Retry: always set `waitBetween` and `maxWait`.** Exponential backoff without bounds can stall your app. Example: `waitBetween = "100ms"`, `maxWait = "2s"`.

6. **Cache: use `@CacheInvalidate(allEntries = true)` sparingly.** Flushing the entire cache on every write doesn't scale. Prefer key-specific invalidation or TTL-based expiration.

7. **Scheduling: check `Thread.interrupted()` in long jobs.** Kora signals shutdown via interrupt; long-running jobs should exit early to avoid delayed shutdown.

8. **Composite cache keys: use records.** `record Key(UUID tenantId, UUID orderId) {}` is clearer than `Object[]` and works out of the box with Caffeine.

9. **Aspect order: outermost annotation runs first.** `@Timeout` outside `@Retry` = timeout caps the entire retry budget. `@Retry` outside `@Timeout` = each attempt gets its own timeout.

10. **Test aspects explicitly.** Write a unit test that triggers the aspect (e.g., force a retry, verify cache miss/hit) — don't assume annotation "just works".
