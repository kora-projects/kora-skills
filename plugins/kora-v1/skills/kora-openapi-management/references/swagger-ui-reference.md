## 1. Overview

Swagger UI provides interactive API documentation visualization. **Recommended** over RapiDoc for production use due to full OpenAPI feature support.

**Requires:** `openapi-management` module + HTTP server (e.g., `http-server-undertow`).

## 3. Configuration

### Minimal Configuration

```hocon
openapi {
    management {
        file = ["openapi/api.yaml"]
        swaggerui {
            enabled = true
            endpoint = "/swagger-ui"
        }
    }
}
```

**Endpoints:**
- `GET /swagger-ui` ŌĆö Interactive UI
- `GET /openapi` ŌĆö Raw spec (single file is served directly at `endpoint`)

### Full Configuration

```hocon
openapi {
    management {
        file = ["openapi/api-v1.yaml", "openapi/api-v2.yaml"]
        enabled = true
        endpoint = "/api/spec"
        swaggerui {
            enabled = true
            endpoint = "/api/docs"
        }
    }
}
```

**Endpoints:**
| URL | Description |
|-----|-------------|
| `GET /api/docs` | Swagger UI with spec selector |
| `GET /api/spec/api-v1` | Raw v1 spec (filename without directory or extension) |
| `GET /api/spec/api-v2` | Raw v2 spec |

## 4. Configuration Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `file` | `List[String]` | ŌĆö | Paths to OpenAPI specs in resources |
| `enabled` | `Boolean` | `false` | Enable the raw-spec endpoints |
| `endpoint` | `String` | `/openapi` | Base path for raw specs |
| `swaggerui.enabled` | `Boolean` | `false` | Enable Swagger UI |
| `swaggerui.endpoint` | `String` | `/swagger-ui` | Swagger UI path |

### Single File Endpoint Behavior

When a single file is specified:
```hocon
openapi.management.file = ["openapi/api.yaml"]
```
Endpoint: `GET /openapi` returns the file content.

### Multiple Files Endpoint Behavior

When multiple files are specified:
```hocon
openapi.management.file = ["openapi/api-v1.yaml", "someDir/api-v2.yaml"]
```
Endpoints:
- `GET /openapi/api-v1` ŌĆö returns `api-v1.yaml`
- `GET /openapi/api-v2` ŌĆö returns `api-v2.yaml` (directory stripped)

The endpoint path is the filename without extension and directories.

## 5. Multiple Specs (Versioning)

Swagger UI automatically provides a dropdown to select between multiple specs:

```hocon
openapi {
    management {
        file = [
            "openapi/api-public.yaml",
            "openapi/api-internal.yaml",
            "openapi/api-admin.yaml"
        ]
    }
}
```

## 6. Environment-Specific Configuration

### Development (Full Access)

```hocon
openapi {
    management {
        file = ["openapi/api.yaml"]
        swaggerui {
            enabled = true
        }
    }
}
```

### Production (Restricted)

```hocon
# application-prod.conf
openapi {
    management {
        enabled = true           # keep raw specs reachable for tooling
        swaggerui {
            enabled = false      # hide the interactive UI in production
        }
    }
}
```

### Environment Variables

```hocon
openapi {
    management {
        enabled = ${OPENAPI_ENABLED:true}
        swaggerui {
            enabled = ${SWAGGER_UI_ENABLED:true}
        }
    }
}
```

## 7. Securing the documentation endpoints

Swagger UI exposes API structure. Restrict access in production.

### Option 1: Disable in production

Override the toggle in a profile config (defaults are already `false`, so this is the production default unless dev enables it):

```hocon
# application-prod.conf
openapi.management.swaggerui.enabled = false
```

### Option 2: Global HTTP interceptor

The management controllers are generated by `OpenApiManagementModule`, so gate them with a single `HttpServerInterceptor` tagged `@Tag(HttpServerModule.class)`. It runs on every request; check the path prefix and let everything else through.

```java
import java.util.concurrent.CompletableFuture;
import java.util.concurrent.CompletionStage;
import ru.tinkoff.kora.common.Component;
import ru.tinkoff.kora.common.Context;
import ru.tinkoff.kora.common.Tag;
import ru.tinkoff.kora.http.common.body.HttpBody;
import ru.tinkoff.kora.http.server.common.HttpServerInterceptor;
import ru.tinkoff.kora.http.server.common.HttpServerModule;
import ru.tinkoff.kora.http.server.common.HttpServerRequest;
import ru.tinkoff.kora.http.server.common.HttpServerResponse;

@Tag(HttpServerModule.class)
@Component
public final class DocsSecurityInterceptor implements HttpServerInterceptor {

    @Override
    public CompletionStage<HttpServerResponse> intercept(Context context,
                                                         HttpServerRequest request,
                                                         InterceptChain chain) throws Exception {
        String path = request.path();
        if (path.startsWith("/swagger-ui") || path.startsWith("/openapi")) {
            if (request.headers().getFirst("Authorization") == null) {
                return CompletableFuture.completedFuture(
                        HttpServerResponse.of(401, HttpBody.plaintext("Unauthorized")));
            }
        }
        return chain.process(context, request);
    }
}
```

A ready-to-copy version with token checking lives in `assets/SwaggerUiSecurityInterceptor.java.template` (and `.kt.template`).

## 8. YAML Configuration

YAML format is also supported:

```yaml
openapi:
  management:
    file:
      - "openapi/api-v1.yaml"
      - "openapi/api-v2.yaml"
    enabled: true
    endpoint: "/openapi"
    swaggerui:
      enabled: true
      endpoint: "/swagger-ui"
    rapidoc:
      enabled: false
      endpoint: "/rapidoc"
```

## 9. Troubleshooting

| Problem | Solution |
|---------|----------|
| 404 on `/swagger-ui` | Check `swaggerui.enabled = true` |
| Empty UI | Verify spec is valid YAML/JSON at `/openapi/{file}` |
| Multiple specs not showing | Each spec needs separate `file` entry |
| UI loads but no endpoints | Check spec has `paths` section |

## 10. Best Practices

1. **Use Swagger UI over RapiDoc** ŌĆö Full support for discriminators, oneOf/anyOf
2. **Disable UI in production** ŌĆö Keep specs for tooling, hide UI
3. **Group related APIs** ŌĆö Single file per API audience (public/internal/admin)
4. **Validate specs** ŌĆö Use `scripts/validate_openapi.py` before deployment
