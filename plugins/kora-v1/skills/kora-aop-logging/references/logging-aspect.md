## Setup

`logging-common` is normally pulled transitively by a backend (`logging-logback`, or `logging-slf4j` with a custom SLF4J binding). Add it explicitly only if missing:

```groovy
implementation "ru.tinkoff.kora:logging-common"
```

```java
@KoraApp
public interface Application extends LoggingModule, /* ... */ { }
```

The aspect is generated by the annotation processor (Java `ru.tinkoff.kora:annotation-processors`) / KSP (Kotlin `ru.tinkoff.kora:symbol-processors`). Without it, the annotations are inert.

## `@Log` family

All annotations are nested types of `ru.tinkoff.kora.logging.common.annotation.Log`.

| Annotation | Effect | Default level |
|-----------|--------|---------------|
| `@Log` | Log on entry and exit | `INFO` |
| `@Log.in` | Log on entry only | `INFO` |
| `@Log.out` | Log on exit only | `INFO` |
| `@Log.result` | Log the return value only | `DEBUG` |
| `@Log.off` (parameter) | Suppress that single argument in the log line | — |
| `@Log.off` (method) | Suppress all logging for the method | — |

`@Log`, `@Log.in`, `@Log.out`, and `@Log.result` are method-level. `@Log.off` targets both methods and parameters. `@Log` targets both methods and parameters too, but on a parameter it is rarely used.

## Choosing the level

The level is the annotation's **`value`** attribute, of type `org.slf4j.event.Level`. There is no `level =` attribute.

```java
import org.slf4j.event.Level;

@Log(Level.DEBUG)
public User getUser(String id) { ... }

@Log.in(Level.TRACE)
public void processOrder(Order order) { ... }
```

`org.slf4j.event.Level` has exactly: `TRACE`, `DEBUG`, `INFO`, `WARN`, `ERROR`. There is **no `OFF`**. To turn off logging for a method, annotate it with `@Log.off` — not a level.

## Output by configured logger level

The annotation level selects *which logger level the entry/exit records are emitted at*. Whether you see data depends on the logger level configured for that class in `logback.xml`.

For `@Log` on `methodWithArgs(String strParam, int numParam)` returning `"testResult"`:

| Logger level | Entry | Exit |
|--------------|-------|------|
| `TRACE` / `DEBUG` | `> {data: {strParam: "s", numParam: "4"}}` | `< {data: {out: "testResult"}}` |
| `INFO` | `>` | `<` |
| `WARN`+ | (nothing) | (nothing) |

`@Log.in` emits only the `>` record; `@Log.out` emits only the `<` record. At `INFO` you get boundary markers without arguments or results — safe for production hot paths. Set the class logger to `DEBUG`/`TRACE` to surface argument/result data.

Control per package/class via `logback.xml`:

```xml
<logger name="com.example.hotpath" level="INFO"/>             <!-- boundaries only -->
<logger name="com.example.service.UserService" level="DEBUG"/> <!-- full data -->
```

## Selective field logging

```java
@Log                                  // log args + result
public Order create(
    CreateOrderDto body,
    @Log.off String idempotencyKey    // never logged
) {
    return ...;
}
```

Use `@Log.off` on a parameter for credentials, tokens, large blobs, and PII. Use `@Log.off` on the method to silence a method entirely:

```java
@Log.off
private void internalCleanup() { ... } // no entry/exit records
```

## Supported signatures

| Java | Kotlin |
|------|--------|
| `T myMethod()` (incl. `Void`) | `fun myMethod(): T` / `T?` / `Unit` |
| `Optional<T> myMethod()` | — |
| `CompletionStage<T> myMethod()` | `suspend fun myMethod(): T` |
| `Mono<T>` / `Flux<T>` (needs `io.projectreactor:reactor-core`) | `Flow<T>` (needs `org.jetbrains.kotlinx:kotlinx-coroutines-core`) |

Java: enclosing class must be non-`final`. Kotlin: enclosing class must be `open`. The aspect generates a subclass.

## Backend / encoder note

The `@Log` aspect writes through SLF4J; `logback.xml` controls the format. Kora ships `ru.tinkoff.kora.logging.logback.ConsoleTextRecordEncoder` for structured text output that renders Kora's MDC and structured arguments. For async output, wrap your appender with `ru.tinkoff.kora.logging.logback.KoraAsyncAppender` — Logback's stock `AsyncAppender` does not propagate Kora's MDC or structured args. See [logging-performance.md](logging-performance.md).

JSON encoders and the full backend setup belong to the SLF4J/Logback module — see `.kora-agent/kora-docs/mkdocs/docs/en/documentation/logging-slf4j.md`.
